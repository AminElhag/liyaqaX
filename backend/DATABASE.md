# DATABASE.md — Backend Database Rules & Standards

This file defines the rules for every database migration, schema decision,
and data access pattern in this project.

No table or column is created without a feature context driving it.
Schema is built incrementally alongside features — never speculatively.

This file is permanent. It is read at the start of every backend development
session alongside `backend/CLAUDE.md`. It does not contain any table
definitions — those live in Flyway migration files.

---

## 1. Migration discipline

### File naming

```
V<version>__<description>.sql
```

- Version is a sequential integer: `V1`, `V2`, `V3` — never decimals (`V1.1`).
- Description is snake_case, concise, describes what changes: `create_members_table`,
  `add_freeze_limit_to_membership_plans`, `create_lead_pipeline_tables`.
- Full example: `V3__add_freeze_limit_to_membership_plans.sql`

### Immutability

- A migration file is **immutable the moment it is applied** to any environment.
- Never edit, rename, or delete a migration file that has been committed to `develop` or `main`.
- Flyway checksums every applied migration. Editing an applied file causes Flyway to refuse
  to start. This is a feature, not a bug.
- To fix a mistake in an applied migration: write a new migration that corrects it.

### Early development exception

During early development (before any staging or production deployment):
- `flywayClean` is permitted in the local dev environment only.
- The `V1__create_schema.sql` file may be edited directly and the database
  recreated from scratch via `./gradlew flywayClean flywayMigrate`.
- This exception ends permanently the moment any data exists in a real environment.
- `spring.flyway.clean-disabled=true` is hardcoded in `application-prod.yml`
  and `application-staging.yml`. It is only `false` in `application-dev.yml`.

### One concern per migration

- Each migration file does one thing: creates a table, adds a column, adds an index,
  modifies a constraint. Do not bundle unrelated changes into one migration.
- Exception: a feature that requires multiple related tables may create all of them
  in one migration if they cannot exist independently (e.g., a table and its join table).

### Backward compatibility (post-launch)

Once deployed to staging or production, every migration must be safe to apply
while the previous version of the application is still running (zero-downtime deploys):
- Adding a nullable column: safe.
- Adding a table: safe.
- Adding an index `CONCURRENTLY`: safe.
- Dropping a column: only after the application code no longer references it.
- Renaming a column: never directly. Add the new column, migrate data, drop the old one
  across three separate deployments.
- Changing a column type: only if the cast is implicit and lossless.

## 2. Standard columns — every table without exception

Every table in this database must have the following columns.
No exceptions. No shortcuts. No "this table is simple, it doesn't need them."

```sql
id          BIGSERIAL       PRIMARY KEY,
public_id   UUID            NOT NULL DEFAULT gen_random_uuid(),
created_at  TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
updated_at  TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
deleted_at  TIMESTAMPTZ     NULL
```

### Why each exists

**`id` (BIGSERIAL)** — internal primary key. Used for all foreign key relationships
inside the database. Never exposed outside the backend. Fast joins, small index.

**`public_id` (UUID)** — the only identifier ever returned in API responses or
accepted as input from any client. UUID v4 generated by PostgreSQL. Per ADR-0001.

**`created_at` / `updated_at` (TIMESTAMPTZ)** — audit trail. `updated_at` is
maintained by a database trigger (see section 9) — application code does not set it.

**`deleted_at` (TIMESTAMPTZ NULL)** — soft delete marker. NULL means the record
is active. A non-null value means deleted. Per ADR-0012. Hard deletes are never
used on business entities. The only tables exempt from `deleted_at` are:
append-only tables (audit_logs, integration_logs, message history) where
records are never deleted by design.

## 3. Tenant scoping

The system uses a shared-schema multi-tenancy model. Per ADR-0005.

### Scoping levels

| Level | Column | Required on |
|---|---|---|
| Organization | `organization_id BIGINT NOT NULL` | Every tenant-owned entity |
| Club | `club_id BIGINT NOT NULL` | Entities scoped below organization |
| Branch | `branch_id BIGINT NOT NULL` | Entities scoped to a specific branch |

### Rules

- Every business entity table that belongs to a tenant carries `organization_id`.
  No exceptions. If you are unsure whether a table needs it, it does.
- Tables scoped to a branch also carry `club_id` and `organization_id` — all three,
  not just `branch_id`. This allows filtering at any level without joins.
- Tenant columns are the first non-standard columns defined after the standard columns.
- Tenant columns always have a `NOT NULL` foreign key constraint referencing the
  parent table's `id` column.
- Every query that touches a tenant-scoped table must include a tenant filter.
  A query without `WHERE organization_id = ?` on a tenant-scoped table is a bug.

### Example

```sql
CREATE TABLE memberships (
    -- standard columns
    id              BIGSERIAL       PRIMARY KEY,
    public_id       UUID            NOT NULL DEFAULT gen_random_uuid(),
    created_at      TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    deleted_at      TIMESTAMPTZ     NULL,

    -- tenant scope (always first after standard columns)
    organization_id BIGINT          NOT NULL,
    club_id         BIGINT          NOT NULL,
    branch_id       BIGINT          NOT NULL,

    -- domain columns follow...
);
```

## 4. Naming conventions

### Tables

- snake_case, plural nouns: `members`, `membership_plans`, `pt_sessions`.
- Join/association tables: `<table_a>_<table_b>` alphabetically:
  `member_pt_packages`, `user_roles`.
- Never abbreviate: `membership_plans` not `mem_plans`, `organizations` not `orgs`.

### Columns

| Pattern | Convention | Example |
|---|---|---|
| Foreign keys | `<referenced_table_singular>_id` | `member_id`, `organization_id` |
| Timestamps | `<event>_at` suffix | `created_at`, `expires_at`, `cancelled_at` |
| Booleans | `is_<state>` or `has_<thing>` | `is_active`, `has_signed_waiver` |
| Monetary amounts | `<description>_halalas` | `price_halalas`, `total_halalas` |
| Localized strings | `<field>_ar`, `<field>_en` | `name_ar`, `name_en` |
| Soft delete | always `deleted_at` | never `is_deleted`, never `deleted` |
| Enum/status | `<noun>_status` | `membership_status`, `payment_status` |
| Counts | `<noun>_count` | `session_count`, `attempt_count` |
| Durations | `<noun>_seconds`, `<noun>_minutes`, `<noun>_days` | `freeze_duration_days` |

### Constraints

```sql
-- Primary key (auto-named by PostgreSQL is fine)
-- Foreign key
CONSTRAINT fk_<table>_<referenced_table>    FOREIGN KEY (col) REFERENCES table(id)
-- Unique
CONSTRAINT uq_<table>_<column(s)>           UNIQUE (col)
-- Check
CONSTRAINT chk_<table>_<description>        CHECK (condition)
-- Not null is inline, not a named constraint
```

### Indexes

```
idx_<table>_<column(s)>
```

Examples:
```sql
CREATE INDEX idx_members_organization_id ON members(organization_id);
CREATE INDEX idx_memberships_expires_at  ON memberships(expires_at);
CREATE UNIQUE INDEX idx_users_email      ON users(email) WHERE deleted_at IS NULL;
```

## 5. Data types

Use the most precise type available. Never use a loose type where a strict one exists.

| Data | Type | Notes |
|---|---|---|
| Internal primary key | `BIGSERIAL` | Auto-incrementing, never exposed |
| Public identifier | `UUID` | Generated by PostgreSQL (`gen_random_uuid()`) |
| Short text (< 255) | `VARCHAR(n)` | Always set a realistic max length |
| Long text | `TEXT` | No artificial length limit |
| Monetary amounts | `BIGINT` | Halalas — never DECIMAL, FLOAT, or NUMERIC |
| Timestamps | `TIMESTAMPTZ` | Always timezone-aware — never plain TIMESTAMP |
| Dates (no time) | `DATE` | Calendar dates only (e.g., a birthday, a plan start date) |
| Boolean | `BOOLEAN` | Never `SMALLINT` or `CHAR(1)` |
| Status / enum | `VARCHAR(50)` + CHECK constraint | See section 6 |
| JSON config | `JSONB` | For flexible config blobs — not for queryable data |
| Counts | `INTEGER` or `BIGINT` | Depending on expected magnitude |
| Percentages / rates | `NUMERIC(5,4)` | For rates like 0.1500 (15%) — not monetary amounts |
| File references | `TEXT` | Store the path/URL — never binary data in the DB |

### What never to use

- `FLOAT` or `DOUBLE PRECISION` — lossy, never appropriate for business data
- `MONEY` — PostgreSQL's money type has locale-dependent behavior
- `TIMESTAMP` without timezone — silently loses timezone context
- `CHAR(n)` — pads with spaces, causes subtle comparison bugs
- `TEXT` for monetary values — use BIGINT
- `BOOLEAN` columns named `deleted` — use `deleted_at TIMESTAMPTZ`

## 6. Enums and status values

Do not use PostgreSQL `ENUM` types. Use `VARCHAR(50)` with a `CHECK` constraint.

**Why:** PostgreSQL `ENUM` types are painful to extend — adding a value requires a
schema migration even in early development. `VARCHAR` + `CHECK` allows adding new
values in a migration without rebuilding the type.

```sql
-- Correct
membership_status VARCHAR(50) NOT NULL
    CONSTRAINT chk_memberships_status
    CHECK (membership_status IN ('active', 'frozen', 'expired', 'terminated', 'pending')),

-- Never do this
membership_status membership_status_enum NOT NULL
```

Document all valid values in a comment on the column:
```sql
-- membership_status: active | frozen | expired | terminated | pending
membership_status VARCHAR(50) NOT NULL
    CONSTRAINT chk_memberships_status
    CHECK (membership_status IN ('active', 'frozen', 'expired', 'terminated', 'pending')),
```

## 7. Foreign keys

- Every foreign key relationship must be explicit with a named constraint.
- Foreign keys always reference the `id` (internal BIGINT) column, never `public_id`.
- Default behavior on delete/update must be explicit — never rely on the PostgreSQL default:

| Relationship | ON DELETE | ON UPDATE |
|---|---|---|
| Child must not exist without parent (cascade delete intent) | `CASCADE` | `CASCADE` |
| Child must be explicitly handled (most business entities) | `RESTRICT` | `CASCADE` |
| Child is optional reference (e.g., created_by user) | `SET NULL` | `CASCADE` |

- Prefer `RESTRICT` as the default. Only use `CASCADE` when you explicitly want
  deleting the parent to delete all children — and document why in a comment.

```sql
CONSTRAINT fk_memberships_member
    FOREIGN KEY (member_id)
    REFERENCES members(id)
    ON DELETE RESTRICT
    ON UPDATE CASCADE,

CONSTRAINT fk_memberships_organization
    FOREIGN KEY (organization_id)
    REFERENCES organizations(id)
    ON DELETE RESTRICT
    ON UPDATE CASCADE,
```

## 8. Indexes

### Always index

- Every foreign key column: `idx_<table>_<fk_column>`
- Every `public_id` column (already covered by the UNIQUE constraint index)
- Every column used in `WHERE` clauses for tenant filtering: `organization_id`, `club_id`, `branch_id`
- Every column used in sort-heavy queries: `created_at`, `expires_at`, `status` columns
- Soft delete filter: partial indexes on `WHERE deleted_at IS NULL` for active-record queries

### Partial indexes for active records

Most queries only want active (non-deleted) records. A partial index on
`WHERE deleted_at IS NULL` is significantly smaller and faster than a full index:

```sql
CREATE INDEX idx_members_organization_id_active
    ON members(organization_id)
    WHERE deleted_at IS NULL;
```

### Composite indexes

When queries consistently filter on multiple columns together, a composite index
outperforms two separate indexes. Column order matters — most selective column first:

```sql
-- Query: WHERE organization_id = ? AND membership_status = ? AND deleted_at IS NULL
CREATE INDEX idx_memberships_org_status_active
    ON memberships(organization_id, membership_status)
    WHERE deleted_at IS NULL;
```

### Never index blindly

Do not add an index "just in case." Every index has a write cost. Add indexes
when a query pattern is known. Add more when profiling reveals slow queries.

## 9. Triggers

### updated_at maintenance

`updated_at` is maintained by a database trigger on every table.
Application code never sets `updated_at` — the database owns it.

Define this function once, reuse it on every table:

```sql
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

Apply to every table:
```sql
CREATE TRIGGER trg_<table>_set_updated_at
    BEFORE UPDATE ON <table>
    FOR EACH ROW
    EXECUTE FUNCTION set_updated_at();
```

This is included in every migration that creates a table.
No table exists without this trigger.

## 10. Soft deletes

Per ADR-0012, hard deletes are never used on business entities.

### How soft delete works

- Deleting a record sets `deleted_at = NOW()`. The row remains in the database.
- All application queries filter `WHERE deleted_at IS NULL` to exclude deleted records.
- This filtering is enforced at the repository layer via a Hibernate Filter —
  application code never adds `AND deleted_at IS NULL` manually.

### What soft delete applies to

All business entities: members, memberships, staff, plans, packages, sessions,
classes, leads, invoices, payments, organizations, clubs, branches.

### What soft delete does NOT apply to

Append-only records that are never deleted by design:
- `audit_logs` — permanent record of every action
- `integration_logs` — permanent record of every API call
- `payments` — financial records are immutable (corrections use refunds)
- `invoices` — financial records are immutable
- `message_history` — messages are archived, not deleted

## 11. Monetary values

Per ADR-0002. Repeated here because violations are silent and dangerous.

- All monetary amounts are `BIGINT NOT NULL`.
- Unit is halalas (smallest SAR unit). 1 SAR = 100 halalas.
- Column names always include the unit: `price_halalas`, `total_halalas`, `balance_halalas`.
- The application layer converts to SAR for display. The database stores only halalas.
- Arithmetic on monetary columns uses integer arithmetic only.
  Never cast to FLOAT for any monetary calculation.
- VAT amounts are stored separately from base amounts:
  `subtotal_halalas`, `vat_amount_halalas`, `total_halalas`.
  `total_halalas = subtotal_halalas + vat_amount_halalas` — always verify this constraint.

## 12. Timestamps and timezones

- All timestamp columns are `TIMESTAMPTZ` — timezone-aware. Per ADR-0013.
- All values are stored in UTC. The database server timezone is UTC.
  Never store Asia/Riyadh local time in the database.
- `created_at` and `updated_at` default to `NOW()` which returns UTC when the
  server timezone is UTC.
- Application code converts UTC to Asia/Riyadh (UTC+3) for display only.
- Date-only columns (birth dates, plan start dates, freeze start dates) use `DATE` type —
  no time component, no timezone ambiguity.

## 13. Localized text columns

The system is bilingual: Arabic and English. Per ADR-0013.

For any user-facing text field that requires both languages:

```sql
name_ar     VARCHAR(255) NOT NULL,  -- Arabic (required)
name_en     VARCHAR(255) NOT NULL,  -- English (required)
```

Both columns are `NOT NULL`. The system never stores a name in one language only.
If one language is not provided at input time, the application layer copies the
provided value to both columns as a fallback — the database never has a null name.

For longer text (descriptions, notes):
```sql
description_ar  TEXT,   -- nullable for optional long-form text
description_en  TEXT,
```

## 14. JSONB usage

`JSONB` is appropriate for:
- Integration configuration blobs (credentials shape varies per integration type)
- Feature flag payloads
- Audit log payloads (before/after state snapshots)
- Any schema-flexible configuration that would otherwise require frequent migrations

`JSONB` is not appropriate for:
- Any field that is queried, filtered, or sorted on regularly — use a proper column
- Any field with a known, stable shape — define explicit columns
- Monetary values — always BIGINT columns
- Status fields — always VARCHAR with CHECK constraint

When using JSONB, add a comment documenting the expected shape:
```sql
-- config: { "apiKey": string, "environment": "sandbox"|"production", "vatNumber": string }
config JSONB NOT NULL DEFAULT '{}'::jsonb,
```

## 15. Comments

Every table has a comment explaining what it represents:

```sql
COMMENT ON TABLE memberships IS
    'An active or historical membership plan instance assigned to a member. '
    'One member may have multiple memberships over time but only one active '
    '(non-deleted, non-expired) membership at any point.';
```

Column comments for any column whose purpose is not immediately obvious:

```sql
COMMENT ON COLUMN memberships.grace_ends_at IS
    'The date after which the member loses access even without renewal. '
    'NULL if the plan has no grace period.';
```

## 16. Migration template

Every new migration file starts from this template:

```sql
-- ============================================================
-- V<N>__<description>.sql
-- <one sentence describing what this migration does and why>
-- ============================================================

-- ── <Domain group name> ──────────────────────────────────────

CREATE TABLE <table_name> (

    -- Standard columns (required on every table)
    id              BIGSERIAL       PRIMARY KEY,
    public_id       UUID            NOT NULL DEFAULT gen_random_uuid(),
    created_at      TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    deleted_at      TIMESTAMPTZ     NULL,

    -- Tenant scope
    organization_id BIGINT          NOT NULL,

    -- Domain columns
    -- ...

    -- Constraints
    CONSTRAINT uq_<table>_public_id
        UNIQUE (public_id),

    CONSTRAINT fk_<table>_organization
        FOREIGN KEY (organization_id)
        REFERENCES organizations(id)
        ON DELETE RESTRICT
        ON UPDATE CASCADE

);

COMMENT ON TABLE <table_name> IS '<description>';

-- Indexes
CREATE INDEX idx_<table>_organization_id
    ON <table_name>(organization_id)
    WHERE deleted_at IS NULL;

-- updated_at trigger
CREATE TRIGGER trg_<table>_set_updated_at
    BEFORE UPDATE ON <table_name>
    FOR EACH ROW
    EXECUTE FUNCTION set_updated_at();
```

## 17. What never belongs in a migration file

- Application logic or business rules (those live in Kotlin service classes)
- Data seeding for production (use a separate seed script or application startup logic)
- `DROP TABLE` statements (use soft deletes — tables are never dropped in production)
- `TRUNCATE` statements (never in a migration — data removal goes through the application)
- Any reference to environment-specific values (URLs, credentials, config values)
- `flywayClean` invocations (this is a Gradle task, never a SQL statement)
